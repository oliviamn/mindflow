how to attach/launch debugger in vscode for python test @pytest.mark.asyncio ?

Debugging pytest tests (including those decorated with @pytest.mark.asyncio) in VS Code involves setting up a proper debug configuration and/or using the VS Code Test Explorer. Below are some common approaches to launch or attach a debugger when running async tests.

1. Use the VS Code Test Explorer (Built-in for Python)

  1. Install the Python extension for VS Code (if you haven’t already).
  2. Open your project/folder in VS Code.
  3. Ensure VS Code recognizes your tests:
    • If you already have pytest installed, VS Code should detect your tests automatically.
    • If not, go to the Command Palette (MacMacMac Cmd+Shift+P, Win/LinuxWin/LinuxWin/Linux Ctrl+Shift+P), and select Python: Configure Tests. Choose pytest and your tests folder.
  4. Once the tests are discovered, you can see them in the Test Explorer (the beaker icon in the side bar) or next to the test functions in the editor.
  5. Set breakpoints in your async test or code under test.
  6. Click the Debug Test icon or right-click the test in the explorer and choose Debug Test.

This will start the test under the VS Code debugger and stop at any breakpoints, even if the test is async.

2. Use a Custom launch.json Configuration

If you prefer a custom debug configuration, you can create or update your .vscode/launch.json:

  1. In VS Code, open the Run and Debug panel (the play icon in the sidebar).

  2. Click create a launch.json file (if none exists).

  3. Add a configuration that invokes pytest. For instance:

    jsonc
    {
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Debug Pytest",
      "type": "python",
      "request": "launch",
      "module": "pytest",
      "args": [
        "-m", "pytest",
        "--maxfail=1", 
        "-q",
        // You can specify specific tests or folders here if needed
      ],
      "console": "integratedTerminal",
      "justMyCode": true
    }
  ]
}

  4. Set breakpoints in your async test or in code being tested.

  5. From the Run and Debug panel, select Debug Pytest and start debugging.

This approach will run all tests via pytest. If you only want to run tests in the current file or a specific file, you can replace the args with something like "${file}" or ["tests/test_example.py"].

3. Using --pdb (Fallback for CLI)

If you just want an interactive debugger in the terminal (not always as comfortable as VS Code’s GUI debugger), you can run:

bash
pytest --pdb path/to/test_file.py

When a test fails (including async tests), this drops you into the standard Python debugger (pdb). While this doesn’t integrate with VS Code’s GUI breakpoints, it can be a quick fallback.

4. Debugging Async Tests with pytest-asyncio

If you are using pytest-asyncio (@pytest.mark.asyncio), the approaches above still work. The Python extension’s debugger and pytest integration handle async tests normally. Just be sure:

  • You have pytest-asyncio installed:
    bash
    pip install pytest-asyncio
  • Your tests are decorated with @pytest.mark.asyncio or use the async_fixture pattern if needed.

Common Pitfalls / Tips

  • Ensure pytest is selected in VS Code’s Python: Configure Tests.
  • If you can’t see the Debug Test button, you might need to restart VS Code after setting up test discovery.
  • If you have multiple Python interpreters, confirm you’ve chosen the correct Python environment in the VS Code status bar (bottom-right corner).
  • If you encounter trouble with breakpoints not hitting, try removing __pycache__ folders or do a clean environment reinstall to ensure a fresh start.

Summary:

  • Easiest approach: Use VS Code Test Explorer to discover and debug tests with breakpoints.
  • More flexible approach: Create a custom launch.json for pytest, then debug with breakpoints set in your test files.
  • CLI fallback: Use pytest --pdb if you’re okay with an interactive console-based debugger.

All of these methods will allow you to debug async (@pytest.mark.asyncio) tests in VS Code.

    bash

  3. Install your project in editable mode:
    bash
    pip install -e .
  4. After this, Python should recognize engine as a top-level package.
