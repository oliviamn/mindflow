以下是一种较为通用、可扩展的 Workflow（工作流） 设计方案，重点围绕 Step 和 Plan 的输入输出接口进行阐述。因为这是一个“可被复用”的工作流系统，Step 和 Plan 需要在不同场景、不同层级下被组合和嵌套。为便于说明，文中可能会举一些示例性的接口或数据结构（可根据实际实现语言和框架进行相应调整）。

1. 核心概念

1.1 Step

  • 最小可执行单元：每个 Step 代表一个原子操作，例如：某个计算任务、外部 API 调用、数据清洗操作等。
  • 可复用：同一个 Step 可以被多个不同的 Plan 所引用，或在同一个 Plan 内多次使用（只要输入输出兼容即可）。
  • 接口：Step 的输入和输出需要具备一致性和兼容性，以便组合形成更大的流程。

1.2 Plan

  • 由 Step 构成的 DAG：Plan 由多个 Step 之间的依赖关系构成一个有向无环图（DAG）。
  • 可以嵌套：Plan 可被当做一个更大的“Step”去复用，也就是说一个 Plan 的输出可以作为另一个 Plan 或 Step 的输入；也可以在一个 Plan 中嵌套另一个 Plan。
  • 接口：Plan 本身也有对外暴露的输入和输出接口，这些接口内部会映射到 Plan 的内部节点（Step）的输入输出。

1.3 为什么需要重点关注输入输出？

  • 连接拼接：Step 与 Step、Plan 与 Plan、Step 与 Plan 之间需要通过输入输出有机拼接。
  • 灵活扩展：可组合的设计意味着系统可以灵活扩展，尤其是当需要做 DAG 优化、并行执行等场景时，能简化数据流和逻辑的管理。

2. Step 的输入输出接口设计

2.1 Step 的抽象接口

可以将 Step 抽象为一个接口或基类（以下用伪代码示例）：

pseudo
interface IStep {
    // Step 的唯一标识符，可用于日志、调度等
    getId(): String 

    // 输入数据结构
    setInput(input: StepInput)

    // 输出数据结构
    getOutput(): StepOutput

    // 执行过程
    execute(context: ExecutionContext): void
}

其中：

  1. StepInput：可定义为一个带有若干字段（Key）的字典或更丰富的类型系统（如使用 JSON Schema、Protocol Buffers 等）。
  2. StepOutput：Step 的输出同理，可以是一个标准格式的字典或其他类型。
  3. ExecutionContext：可包含执行过程所需的上下文信息，例如日志接口、全局配置、执行环境上下文等。

要点：

  • Input 和 Output 要有 类型定义，这样可以在编译/注册/执行时做校验。
  • Output 也可以在 execute() 函数执行后才被计算并填充。
  • 某些场景下，Input 和 Output 也可以仅仅是 Key-Value 格式，以最大化通用性。但需要配合额外的类型约束或断言。

2.2 Step 的输入输出兼容

在一个 DAG 中，如果要让 Step B 接在 Step A 之后执行，需要 B 的输入与 A 的输出相匹配，或有某种可转换的策略。例如：

  • 字段名和类型 能够一一映射
  • B 的输入是一个子集/或超集，或者可以通过一些转换函数自动转换

这就要求系统可能需要：

  • 在注册 Step 时，声明其所需的输入类型与输出类型
  • 在系统构建 DAG 时，对各 Step 节点的连接进行合法性检查（或在运行时动态判断）

3. Plan 的输入输出接口设计

3.1 Plan 的抽象接口

Plan 可以看作“更高层次的 Step”，也需要有类似的接口定义，但内部是一个 DAG。

pseudo
interface IPlan {
    // Plan 的唯一标识符
    getId(): String 

    // Plan 输入
    setInput(input: PlanInput)

    // Plan 输出
    getOutput(): PlanOutput

    // 构造和管理 DAG
    addStep(step: IStep): void
    addEdge(stepA: IStep, stepB: IStep): void
    // ... 其他管理 DAG 的方法 ...

    // 执行入口
    execute(context: ExecutionContext): void
}

与 Step 的差异在于：

  1. PlanInput 和 PlanOutput 一般也需要定义一个特定的结构/类型。
  2. Plan 内部管理一批 Step，并且需要定义它们的依赖顺序（DAG）以及输入输出的映射规则。

3.2 Plan 的输入输出映射

为了让 Plan 也能像 Step 一样拼接到更大的流程里，我们需要解决：

  1. Plan 的输入 如何传递给内部的第一个或多个 Step？
  2. Plan 的输出 如何从多个内部 Step 的输出中聚合或选取？

最常见的方式是：

  • Plan 的输入：由外部传入后，内部会将其分发给指定的一些 Step 的输入。可以是 1 个 Step，也可以是多个 Step 并行启动。如果 Plan 有多个输入 Key，则需要能够分别映射到不同的 Step 里。
  • Plan 的输出：执行完成时，可能需要从 Plan 中的一个或多个 Step 的输出里取到结果，组合形成 Plan 的对外输出。可以是：
    • 选取某个最终 Step 的输出作为 Plan 输出
    • 组合多个 Step 的部分输出并聚合

要点：

  • 在定义 Plan 时，要 明确 哪些 Steps 需要接收 PlanInput，以及如何整合输出形成 PlanOutput。
  • 如果 Plan 是嵌套在更高层级的 Plan 或者要与其他 Step 联动，需要把输入输出声明清晰，避免数据冲突或类型不匹配。

4. Plan 嵌套构建的思路

4.1 复用 Plan

由于 Plan 是一个带输入输出的可执行单元，它可以被当做更高层次的“Step”被复用。例如：

pseudo
Plan bigPlan = new Plan();
Plan subPlan1 = new Plan();
Plan subPlan2 = new Plan();

// 先在 subPlan1 和 subPlan2 中定义好 DAG
// ...

// bigPlan中，subPlan1 和 subPlan2 可以被当做 step
bigPlan.addStep(subPlan1);
bigPlan.addStep(subPlan2);

// 构建 bigPlan 的执行逻辑
bigPlan.addEdge(subPlan1, subPlan2);

4.2 建立输入输出映射

当我们把一个 Plan 当做一个 Step 使用时，需要指定：

  • subPlan.setInput(inputMapping)：将外部数据映射到 subPlan 的输入；
  • subPlan.getOutput()：决定如何读取 subPlan 的输出并传递给下一个节点（Step 或 Plan）。

5. 其他需要注意的要点

5.1 并行与调度

  • DAG 结构：当某些 Step 不存在数据依赖时，可以并行调度执行，以提升效率。
  • 调度器：需要一个调度器（Scheduler 或 orchestrator）来解析 DAG，判断哪些 Steps 当前可以执行、哪些需等待依赖完成。
  • 资源控制：有时需要对并行任务的数量做限制，以及负载均衡、排队机制等。

5.2 错误处理与重试

  • 当某个 Step 执行失败时，是否需要整个 Plan 停止？能否局部重试？
  • 是否会有状态回滚（类似数据库事务）或者补偿机制？
  • 在错误重试后，是否要保留中间产物？这些都需要在设计时加以考虑。

5.3 版本化与可追溯

  • 版本化：Step 的实现可能会持续迭代，如何保证新旧 Step 的兼容或回溯执行？
  • 可追溯：在大规模数据处理或任务编排中，需要可以追踪每一个 Plan 的执行情况、日志信息，以及产出的可视化拓扑图。

5.4 配置驱动/DSL

  • 是否需要用一个 DSL（领域特定语言）或配置文件来描述整个 Plan 的 DAG？
  • 这样可以让系统更易于扩展和管理，也便于把 Plan 的定义从代码中分离出来，支持可视化编辑、自动化生成等。

5.5 安全与权限

  • 如果某些 Step 需要访问敏感数据或外部系统 API，是否需要鉴权？
  • 不同角色在编辑/执行 Plan 时权限是否相同？

5.6 监控与日志

  • 大规模执行时，需要完整的监控指标（执行时间、资源占用、吞吐量等）和日志追踪。
  • 方便进行诊断和性能优化。

5.7 数据格式/Schema 演化

  • 如果输入输出中涉及到复杂数据结构，随着业务迭代可能会出现字段变动，需要考虑 Schema 演化、兼容、升级。

6. 总结与示例性数据结构

6.1 总结

  1. Step：最小可执行单元，需定义清晰的 Input/Output（可能是 KV 或强类型）和执行逻辑。
  2. Plan：由多个 Step（或子Plan）构成一个 DAG，并提供对外的接口，用于输入映射与输出聚合。
  3. 互相拼接：Step 可以连接到 Step，Plan 可以包含 Step，也可以包含其他 Plan；Plan 的输出可以作为下一个节点（Step 或 Plan）的输入。
  4. 其他关注点：并行调度、错误重试、可追溯与监控、版本化、DSL 驱动等，都需要在系统设计时一并考虑。

6.2 示例：数据结构（可参考）

假设我们用 JSON 来描述输入输出：

StepInput / StepOutput

json
{
  "type": "object",
  "properties": {
    "fieldA": { "type": "string" },
    "fieldB": { "type": "number" },
    "fieldC": { "type": "object" }
  },
  "required": ["fieldA"]
}

Step

pseudo
class ExampleStep implements IStep {
    private StepInput input;
    private StepOutput output;
    private String id;

    // constructor, getters, setters ...

    public void execute(ExecutionContext context) {
        // 在此进行实际业务逻辑
        // 例如：将 input 中的一些数据做计算，或发起 API 调用
        // 最后生成 output
    }
}

Plan

pseudo
class ExamplePlan implements IPlan {
    private List<IStep> steps;
    private DAG dag;
    private PlanInput input;
    private PlanOutput output;
    private String id;

    // 构建 DAG 的方法
    public void addStep(IStep step) {
        steps.add(step);
        dag.addNode(step);
    }

    public void addEdge(IStep a, IStep b) {
        dag.addEdge(a, b);
    }

    public void execute(ExecutionContext context) {
        // 1. 将 PlanInput 映射到对应的 StepInput
        // 2. 调用调度器执行 DAG（可能并行执行一些 step）
        // 3. 收集最后需要的输出，合并成 PlanOutput
    }
}

7. 结语

以上是一个较为通用的 Workflow 设计思路，主要围绕 Step 和 Plan 的输入输出接口，及其在 DAG 中的组合方式。核心在于：

  1. 抽象清晰：Step 与 Plan 都有相对固定的接口定义（输入、输出、执行逻辑）。
  2. 可组合性：Plan 能作为更高层次的“Step”被嵌套复用；Step 之间也能灵活地用其输出对接下一个 Step 的输入。
  3. 可扩展性：在此基础上，可以随业务需要增加并行调度、错误处理、日志监控、版本管理等特性。

希望上述思路能对您搭建 Workflow 系统有所帮助。具体实现细节需结合实际的语言、框架和业务需求进行调整。若有更多问题，可根据场景做进一步探讨。

You said:
若以python作为开发语言，input类型为Dict[str, Any], output类型为　Dict[str, Any],  在执行时，iterate input的key, 从前一Step的output中取对应key的值，是否可行？是否有更好的建议？

简短回答：
直接对每个 Step 使用 Dict[str, Any] 作为输入和输出，通过遍历输入的 key，在前一个 Step 的 output 中取对应值，这种做法是可行的。它简单直接，在原型或较小型的流程中常被使用。但若系统规模和复杂度提升，或需要更严格的类型校验/调度管理，建议配合**更明确的“映射关系”或强类型（如 Pydantic、数据类）**来使输入输出更清晰、更易维护。

1. 直接使用 Dict[str, Any] 的可行性

  1. 简单直接：

    • Python 对字典操作非常灵活，直接“取值、赋值”就可以连接前后 Step；
    • 写起来快速且便于原型验证（prototype）。

  2. 对小型流程足够：

    • 如果你的业务流程不多、数据结构相对稳定，用字典简单拼接就够了；
    • 当出现 KeyError 或者类型不匹配时，很容易在小范围内发现问题。

  3. 缺点/风险：

    • 缺乏可读性：随着流程增多、字典字段复杂，一旦修改了某个字段名，就需要逐处查找、改动；
    • 缺少静态或半静态校验：只能在运行时发现 KeyError 或类型错误；
    • 调试和维护成本上升：当多个 Step 的输入输出 key 相互交叉，需要做大量人工维护或写一堆映射逻辑。

因此，如果只是做一个轻量、简单的自动化脚本，或者一个小规模的工作流，遍历 key 来做字典映射没有问题。但在更严谨或更大型的项目里，需要在此之上增加一些数据流管理和类型校验的机制。

2. 如何改进？（更好的建议）

以下是几个常见的改进思路，可根据业务复杂度和团队习惯进行选择或组合使用。

2.1 显式的“字段映射”配置

  • 思路：在两个 Step 之间，不仅仅是“遍历 key 并复制”，而是先明确列出输入字段 -> 输出字段的对应关系（mapping）。
  • 配置方式：
    1. 通过一个 Python 字典/JSON/YAML 来维护映射，示例：
      yaml
      step_mapping:
  - from_key: "inputA"
    to_key: "outputX"
  - from_key: "inputB"
    to_key: "outputY"
    2. 通过 Python dataclass 或者类属性注解，让开发者显性声明“我需要从哪个字段取值”。
  • 优点：
    • 任何字段改名，都只需改 mapping 配置；
    • 便于自动化或可视化展示；
    • 当流程很大时，可以以表格或配置文件方式查看数据流向。
  • 缺点：
    • 需要写更多的“映射配置”，在小型流程中可能觉得繁琐；
    • 如果字段非常多，并且出现一一对应的情况，会略显冗余。

2.2 使用“强类型”或“模型”来表示输入输出

  • 思路：替代“Dict[str, Any]”这种弱类型方式，使用 Python 强类型工具（例如 pydantic 或 dataclasses）定义输入输出模型。

    python
    from pydantic import BaseModel

class StepInputModel(BaseModel):
    name: str
    value: int

class StepOutputModel(BaseModel):
    result: float
    status: str

  • 优点：

    • 在传入时会自动做类型校验和数据验证，减少“潜在的 KeyError、类型错误”；
    • 开发时 IDE 会有更好的自动补全和类型提示，代码可读性与可维护性更好；
    • 当需求变更时，只需在模型层更新字段即可，能减少上下游不同步的问题。

  • 缺点：

    • 在一些原型或极简场景下，写数据模型增加了额外的“定义步骤”；
    • 需要团队对 Pydantic 或 dataclasses 等库有一定熟悉度。

2.3 中央化的“数据路由（router）”或“上下文（context）”机制

  • 思路：在执行 Workflow 时，给每个 Step 传递的并不一定仅仅是 dict，而是一个拥有 get/set 方法的上下文对象，由系统内部来管理真实的数据存储和映射：
    • 例如 context.set_value("step_a.result", 100) 或 context.get_value("step_b.input")；
    • 这样可以统一管理所有 Step 之间的数据传递，避免直接在 Step 内到处写“dict['keyA'] = ...”。
  • 优点：
    • 能做额外的功能，如日志记录、数据版本控制、访问权限控制；
    • 可以在一个集中地方处理所有字段映射、兼容检查等逻辑。
  • 缺点：
    • 需要额外实现 context/路由对象，相当于把“dict 取值赋值”这件事封装到一个中间层；
    • 对小规模流程来说可能显得有点“杀鸡用牛刀”。

2.4 混合使用“弱类型 dict” + “可选校验”

  • 思路：依旧保持 Step 接口的 input: Dict[str, Any] -> output: Dict[str, Any]，但在必要时提供可选的schema检查或动态校验：
    1. 在 Step 内部，用 Pydantic Model 或自定义校验函数对 input 做一次校验；
    2. 在输出时，对生成的 dict 做一次校验。
  • 优点：
    • 在开发时保持使用 dict 的灵活性；
    • 在关键位置提供校验，及时发现数据不匹配的问题；
    • 可以仅在开发阶段、测试阶段或特定环境下启用校验，生产环境可以切换成无需校验的快速模式。
  • 缺点：
    • 还是需要写一部分校验逻辑；
    • 如果多次嵌套使用 dict，可能出现嵌套校验比较麻烦。

3. 实际示例

以下示例展示了最简单的实现方式（dict 对 dict），以及如何稍微改进一下让其更可维护。

3.1 最简单的纯字典传递

python
def step_a(input_data: Dict[str, Any]) -> Dict[str, Any]:
    # 假设 input_data = {"a": 1, "b": 2}
    # 做点什么操作
    result = input_data["a"] + input_data["b"]
    return {"sum_result": result}

def step_b(input_data: Dict[str, Any]) -> Dict[str, Any]:
    # 假设 input_data = {"sum_result": 3}
    return {"double_result": input_data["sum_result"] * 2}

# workflow 逻辑
output_a = step_a({"a": 1, "b": 2})
output_b = step_b(output_a)
print(output_b)  # {"double_result": 6}

特征：

  • 快速、简单；
  • 当字段名改变，需要到处修改；
  • 一旦漏了 Key，运行时才报错。

3.2 使用“映射函数”封装

python
def map_fields(source: Dict[str, Any], mapping: Dict[str, str]) -> Dict[str, Any]:
    # 根据 "输入字段 -> 输出字段" 关系做转换
    result = {}
    for from_key, to_key in mapping.items():
        result[to_key] = source[from_key]
    return result

def step_a(input_data: Dict[str, Any]) -> Dict[str, Any]:
    result = input_data["a"] + input_data["b"]
    return {"sum_result": result}

def step_b(input_data: Dict[str, Any]) -> Dict[str, Any]:
    return {"double_result": input_data["sum_result"] * 2}

mapping_a_to_b = {"sum_result": "sum_result"}

# workflow
out_a = step_a({"a": 1, "b": 2})
# 当step_b需要的input字段和step_a输出字段不一致时
# out_b_input = map_fields(out_a, mapping_a_to_b)
out_b_input = out_a  # 如果正好同名，就可以直接传
out_b = step_b(out_b_input)

特征：

  • 显式维护“字段对应关系”可以减少硬编码依赖；
  • 适合更复杂的场景，比如“fieldA -> fieldX, fieldB -> fieldY”。

3.3 使用 Pydantic 做输入输出模型

python
from pydantic import BaseModel, Field

class StepAInput(BaseModel):
    a: int
    b: int

class StepAOutput(BaseModel):
    sum_result: int

class StepBInput(BaseModel):
    sum_result: int

class StepBOutput(BaseModel):
    double_result: int

def step_a(input_data: Dict[str, Any]) -> Dict[str, Any]:
    # 校验输入
    validated_input = StepAInput(**input_data)
    # 执行业务逻辑
    result = validated_input.a + validated_input.b
    # 校验并返回输出
    output = StepAOutput(sum_result=result)
    return output.dict()

def step_b(input_data: Dict[str, Any]) -> Dict[str, Any]:
    validated_input = StepBInput(**input_data)
    double_val = validated_input.sum_result * 2
    output = StepBOutput(double_result=double_val)
    return output.dict()

# workflow
out_a = step_a({"a": 1, "b": 2})       # => {"sum_result": 3}
out_b = step_b(out_a)                 # => {"double_result": 6}

特征：

  • 能在调用时做严格的“字段和类型”检查，避免在运行时出现不可预知的错误；
  • 当字段名或类型发生变化，只需改对应的 Pydantic 模型；
  • 适合较复杂、可扩展的场景。

4. 结论

  • Iterate dict key 取值：

    • 可行，尤其在小规模、快速原型或演示中十分常用；
    • 但随着业务复杂度提升，手动维护字段名和类型可能会让代码难以阅读、维护、调试。

  • 更好的实践：

    1. 映射配置：在需要灵活多变的字段映射时，使用一个“字段映射”的配置，避免硬编码；
    2. 强类型/模型：通过 Pydantic 或自定义数据类进行输入输出校验，增强可读性与可靠性；
    3. 上下文路由：在工作流编排层面，使用统一的上下文对象、调度器来分发数据，简化 Step 间的直接依赖。

总之，如果只是 Demo 或小项目，“字典 + key 遍历取值”足以。但若打算将其发展成一个可维护、可扩展的工作流系统，则需要尽早在设计中加入数据映射或类型校验的思路，以避免后期维护成本过高。

下面给出一个示例，演示在一个典型的 Workflow Engine 场景中，前后 Step 的执行以及数据如何进行传递的基本设计思路。示例中会体现一个 Context 与 Executor/调度器 的概念，展示它们如何在工作流执行的过程中配合完成数据的传递与步骤的执行。

1. 核心角色

  1. Step：

    • 工作流中最小的可执行单元，描述了具体要做的事情（任务逻辑）。
    • 每个 Step 有自己的输入/输出需求，但并不直接操作前后 Step 的数据。
    • 通过 Context 或者别的机制获取输入数据，并将结果写回 Context。

  2. Plan / Workflow：

    • 由多个 Step 组成的有向无环图（DAG）。
    • 描述了 Step 间的依赖和连接关系。

  3. Executor / Scheduler：

    • 负责解析 Plan（DAG），决定执行顺序（或并行执行）并管理执行过程。
    • 调用各个 Step 的 run() 方法，传入/回收数据。

  4. Context：

    • 在工作流执行时，保存和传递上下文数据，包括每个 Step 的中间输出、全局配置信息等。
    • Step 执行时，通过 context.get_value(...) 读取前置步骤的数据，通过 context.set_value(...) 写入自身输出。

2.4 WorkflowPlan (DAG)

python
class WorkflowPlan:
    """
    代表一个由多个 Step 构成的工作流（DAG）。
    这里用非常简化的方式来管理 Step 依赖关系。
    """
    def __init__(self):
        self.steps = []               # List[Step]
        self.dependencies = {}        # { stepA: [stepB, stepC], ... } 表示 A -> B, A -> C

    def add_step(self, step: Step):
        self.steps.append(step)
        self.dependencies[step] = []

    def add_dependency(self, predecessor: Step, successor: Step):
        """
        predecessor 执行后，successor 才能执行
        """
        self.dependencies[predecessor].append(successor)

    def topological_sort(self) -> "List[Step]":
        """
        简化示例：进行一次 Topological Sort 并返回按顺序排列的 Step 列表
        """
        # 这里只是个伪实现，真实场景需要完整的 DAG 拓扑排序
        # ------------------------------------------------
        in_degree = {step: 0 for step in self.steps}
        for step, successors in self.dependencies.items():
            for succ in successors:
                in_degree[succ] += 1

        queue = [s for s in self.steps if in_degree[s] == 0]
        result = []

        while queue:
            current = queue.pop(0)
            result.append(current)
            for succ in self.dependencies[current]:
                in_degree[succ] -= 1
                if in_degree[succ] == 0:
                    queue.append(succ)

        return result

        numbers = data["numbers"]

        compute_result = context.get_value("compute_result")

        # 模拟保存操作，这里仅打印

4. 设计要点与扩展

  1. Context 作为数据传输中心

    • 每个 Step 不直接操作其它 Step 的输出，而是通过 Context 进行读写；
    • 这样就有了一个统一的数据管理位置，易于对数据访问进行权限限制、日志记录或版本化等扩展。

  2. Executor / Scheduler 管理执行

    • WorkflowExecutor 负责从 WorkflowPlan 获取执行顺序，处理并行、重试、失败时回滚等；
    • 也可结合队列/线程池/分布式调度进行扩展。

  3. 异常处理 & 重试机制

    • 在真正的 Workflow Engine 中，很多 Step 可能外部依赖（数据库、API 等），常常需要重试策略（例如 3 次重试、指数退避等）。
    • 可以在 executor.execute() 中捕获异常后，选择重试还是跳过或终止。

  4. 日志与监控

    • 可在每次 step.run(...) 前后打日志、记录执行时间和错误原因。
    • 也可将 Context 中的数据进行序列化，做数据血缘追溯。

  5. 并行执行

    • 如果多个 Step 互不依赖，可以在 DAG 调度时并行执行；
    • 具体可以在 Executor 中使用多线程 / 多进程 / 分布式任务队列等方式实现。

  6. Plan 嵌套

    • 将一个复杂的 Plan 封装成可复用的“子Plan”，它对外只暴露输入与输出，内部仍是一个 DAG；
    • 子Plan 本身可以作为一个 “Step” 放到更大的工作流里。

5. 小结

上面的示例展示了一个典型 Workflow Engine 的核心设计：

  • Context 作为数据存取中心；
  • Executor（或 Scheduler）负责按照 DAG 的顺序调度执行；
  • Step 只关注自身逻辑，从 Context 中读入所需数据，再将结果写回 Context；
  • Plan（DAG）描述 Step 之间的依赖关系。

这套思路非常常见于实际的工作流/调度系统（如 Apache Airflow、Luigi、Prefect 等），只是在实际工业级产品中，会有更多的特性：

  • 并行调度、分布式执行、可视化界面、参数化运行、版本管理、运维监控等。

但在本示例里，核心结构已经体现了工作流引擎最基本的概念和数据流传递方式。
